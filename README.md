train.csvに含まれる以下のような問題も修正しました
- PDFのOCRのミス(転写文または翻訳文が全く別物になっている)
- コンペのホストが提供したtrain.csvで、転写文がtruncatedされていたものを復元(これはpublished_texts.csvから比較的簡単に復元できました)

以下のような問題も至る所で見つかりましたが、あまりに分量が多いのと、おそらく最終評価で使うtest.csvにも同じような問題があると考えられたので、あえてそのままにしました
- PDFにおいて斜体の文字がOCRで書き出せていない

なお、PDFを使ったtrain.csvの修正作業については、該当箇所のスクリーンショットを撮って、ChatGPTに文字起こしさせるという原始的な方法を取りました。

*文字起こし用*
````markdown
# 目的（あなたへの依頼）
これから送る**スクリーンショット画像**には、左（または上）に**アッカド語の転写（transliteration）**、右（または下）に**英語訳（translation）**が並んでいます。  
画像からそれぞれを**正確に文字起こし**して、**スプレッドシートにコピペしやすい形**で返してください。  
場合によっては一枚のスクリーンショットに収まらず複数枚のスクリーンショットに分かれるかもしれません。その場合はスクリーンショットの並び順を推測して書き起こして下さい。

---

## 返答フォーマット（コピペ最優先）
原則、毎回この2行セットで返してください（余計な説明は不要）:

transliteration(アッカド語本文) <改行>
<区切り>
translation(英訳本文)

- <改行>はChatGPTのUIで改行されるという意味で、<区切り>も同様にChatGPTのUIで改行されるという意味。
- アッカド語の本文と英訳本文はそれぞれ**1行ずつ**に収める

---

## 本文と注記の扱い（重要）
- 画像中に「本文」以外（注釈・見出し・版面注記・ページ番号・行番号など）が混ざっていることがあります（例: `Seal A`, `rev.`, `obv.`, `(remainder obv. missing)`, `(3.5 cm un-inscribed)` など）。
- **文字起こしは本文だけを抽出**し、こうした**注記っぽい行は本文から除外**してください。

出力例（NOTE がある場合）:
アッカド語：書き起こしたtransliteration(本文) <改行>
<改行>
英語：書き起こしたtranslation(本文)


---

## 文字起こしのルール（重要）
### 1) 置き換え（必須）
欠損や破損を表す表現は、意味が同じなら**すべて `<gap>` に統一**してください。

- `x` → `<gap>`
- `[x]` → `<gap>`
- `…`（三点リーダ/省略記号）→ `<gap>`
- `(break)` → `<gap>`
- `(large break)` → `<gap>`
- `(n broken lines)` → `<gap>`（n は数字でも `<gap>` にしてOK）
- `<gap> <gap>`のように`<gap>`が複数回連続した場合は一つにまとめる。つまり`<gap> <gap>` → `<gap>`
- **その他の欠損/破損/読めない旨全般** → `<gap>`

決定詞（determinatives）は次のように統一してください。

- `(d)` → `{d}`
- `(ki)` → `{ki}`

括弧付きの語の扱い：

- `(TÚG)` → `TÚG`（括弧を外す）

---

## 2) そのまま維持してほしいもの
- 大文字小文字、アポストロフ、ハイフン、下付き数字（例: `₂`）、特殊文字（例: `š ṣ ṭ ḫ`）、アクセント等は**見たまま維持**
- 記号や区切り（`-` や `.` など）がある場合も、**画像通り**に（上の置き換えルール対象を除く）
- `<gap>` は必ず **小文字**で `<gap>` とする

---

## 3) 不確実性の扱い
- どうしても判別できない1文字/1語がある場合は、その部分を `<gap>` にしてよい
- ただし、読めるところは極力そのまま残す（全部を `<gap>` にしない）

---

## 画像の前提（コンペ文脈の最小）
- 入力: アッカド語（楔形文字の転写）＝ transliteration
- 出力: その英語訳＝ translation
- ここで作りたいのは「転写 → 英訳」のペアなので、**2列のテキスト**として抜き出すのが目的です

---

## 作業を速く正確にするためのお願い
- 可能なら、画像内の「転写部分」と「英訳部分」をそれぞれ見て、**混ざらない**ように抽出してください
- 句読点や括弧の閉じ忘れが起きやすいので、**左右（または上下）で対応が崩れていないか**を軽く確認してから出力してください
- 出力はスプレッドシート貼り付けが目的なので、**余計な装飾（箇条書き・引用など）なし**でお願いします
````

*長いテキストのアライメント分割用*
````markdown
# 目的（あなたへの依頼）

これから送るテキストとスクリーンショット画像は、Kaggle コンペ **Deep Past Challenge - Translate Akkadian to English** の学習データ候補です。  
1 件の長いレコードについて、**アッカド語の転写（transliteration）** と **英語訳（translation）** の対応を崩さないように注意しながら、**少数（2〜5個程度）の独立レコード**へ分割してください。

この作業の目的は、ByT5 モデルで扱いやすいように、各レコードを **1024 byte 未満**に収めることです。  
ただし、**分割数を増やしすぎて対応関係を壊すくらいなら、分割不能として返す**ほうを優先してください。

---

## 最重要方針（v2: 分割を細かくしすぎない）

- 目標分割数は **2〜5**（可能なら **2〜4** を優先）
- 分割は **必要最小限**にする
  - 追加の分割境界は「1024 byte 未満に収めるために本当に必要」または「対応関係を守るために必要」な場合だけ入れる
- 1チャンクを短くしすぎない
  - byte を厳密計数できない場合は安全側に倒しつつも、**極端に短いチャンクを量産しない**
  - 目安として、可能なら各チャンクは **大きめ（概ね 700〜1000 byte 程度）**を狙う
- もし 5 分割でも収まらない/対応が崩れるなら `needs_manual_review` を返す（無理に 6 個以上へ増やさない）

---

## 背景（最小限）

- このコンペでは、アッカド語の転写から英語訳を生成するモデルを学習します。
- ByT5 は byte-level のモデルなので、文字数ではなく **UTF-8 byte 長**が効きます。
- source 側では通常 `translate Akkadian to English: ` という prefix が付きます。
- 実務上、1 レコードあたり source / target ともに **1024 byte 未満**に抑えたいです。
- 正確な byte 数をあなたが毎回厳密計算する必要はありません。厳密計数に自信がない場合は、**安全側に倒して少し短め**にしてください。

---

## あなたにやってほしいこと

1. 与えられた `transliteration` と `translation` を読む
2. 必要ならスクリーンショット画像も参照する
3. **意味的に対応している境界**を少数だけ見つける（入れすぎない）
4. 1 件を **2〜5** の `transliteration_chunk` / `translation_chunk` に分ける（最小分割を選ぶ）
5. 分割が危険なら `needs_manual_review` と返す

---

## 絶対ルール

### 1) これは「分割」であり「修正」ではありません

- `transliteration` と `translation` の本文は、**原則として入力テキストの部分文字列をそのまま使ってください**
- 勝手な補完、意訳、要約、翻訳し直しは禁止です
- OCRミスや不自然な英語が見えても、**このタスクでは直さない**でください

### 2) 順序を保存してください

- 分割後のチャンクを上から順に並べると、元の `transliteration` / `translation` の順序が維持されている必要があります

### 3) 対応が見える境界でしか切らないでください

- 英語側の文や節の区切り
- 品目列挙の区切り
- 書簡定型句の切れ目
- 命令文・条件文の切れ目
- 転写側の明確な行まとまり

このような **両側でまとまりが見える場所**だけで切ってください。

### 4) 次のものは途中で切らないでください

- 数値、分数、小数、税率
- 銀量や shekel / mina のまとまり
- 人名、神名、地名
- `son of ...` や `belonging to ...` のまとまり
- 引用符の内側の途中
- 品目名の途中

### 5) 無理なら保留してください

次のような場合は、無理に分割せず `needs_manual_review` を返してください。

- 画像を見ても対応境界が確定できない
- train の英訳が途中で壊れているように見える
- 原文と英訳がずれていて 1:1 対応が不明
- 2〜5 分割に抑えようとすると、どう切っても数値列や固有名詞列を破壊してしまう

---

## 画像の使い方

スクリーンショット画像は、**どこで切ると transliteration と translation の対応が自然か**を判断するために使ってください。

重要:

- PDFに写っているアッカド語と英訳文は、チャットで直接渡される `transliteration` / `translation` のテキストと**表記が一致しないことがあります**
- これは `<gap>` のような置き換え、記号整理、その他の整形が、チャットで渡すテキスト側に入っている可能性があるためです
- したがって、**PDFは対応境界を見つけるための参考資料**として使い、最終的な出力本文は **チャットで直接渡されたテキストを優先**してください
- PDFの見た目に合わせて、チャットで渡された本文を別表記へ書き換えないでください

画像を使うときの優先順位:

1. `transliteration` と `translation` の対応境界を確認する
2. train テキストだけでは曖昧な切れ目を補う
3. どこからどこまでが同じ意味のまとまりかを判断する

画像を使っても **新しい翻訳文を作る**ことはしないでください。

---

## どこで切るとよいかの目安（少数分割向け）

### 切れ目候補（優先度高）

- `a-na ... qí-bi-ma um-ma ...` のような導入句の後（ただし短すぎるなら導入句だけで切らない）
- `all this ...`, `thereof ...`, `therefore ...`, `if ... then ...` など「段落・節が切り替わる」英訳表現
- 長い品目列挙の「大きなまとまり」が切り替わる場所（小項目ごとに切りすぎない）
- 同一構文の反復がまとまって終わる場所

### 切れ目候補（やりがちだが禁止に近い）

- 英訳の 1 文ごとに切る（分割が増えすぎやすいので避ける）
- 短い列挙項目ごとに切る（避ける）

---

## 出力形式（厳守）

**JSON 配列だけ**を返してください。余計な説明文は不要です。

### 分割できる場合（2〜5個）

```json
[
  {
    "record_id": "元レコードID",
    "split_id": "元レコードID__01",
    "status": "ok",
    "transliteration_chunk": "ここに分割後の転写本文（入力の部分文字列そのまま）",
    "translation_chunk": "ここに対応する英訳本文（入力の部分文字列そのまま）",
    "boundary_reason": "なぜこの境界で切ったか（対応の根拠を1文）",
    "confidence": "high"
  }
]
```

### 分割が危険な場合

```json
[
  {
    "record_id": "元レコードID",
    "split_id": "元レコードID__00",
    "status": "needs_manual_review",
    "transliteration_chunk": "",
    "translation_chunk": "",
    "boundary_reason": "2〜5 分割に抑えつつ対応を壊さず切れないため",
    "confidence": "low"
  }
]
```

---

## `confidence` の目安

- `high`: 画像とテキストの両方から、対応境界がかなり明確
- `medium`: ほぼ妥当だが、片側の対応が少し弱い
- `low`: train 原文の欠落やズレが疑われる

---

## 実際の依頼を受けたら守ること

- JSON 配列以外を返さない
- 2〜5 分割を強く優先し、境界を増やしすぎない
- 分割であって修正ではないことを忘れない
- 画像を見ても、本文にない語を補わない
- 迷ったら `needs_manual_review` を返す
````

# 最終サブミットのパイプライン
パブリックノートブック「LB 35.9 Ensembling & Post Processing Baseline」をベースにしつつ、使用するモデルを増やしたり、前処理後処理を少し変えたりしました。
最終サブミットの二つは、モデルの学習のエポック数が異なるだけで、過学習を懸念されながらもエポック数を多くしたものの方がプライベートLBスコアが高かったです。
## 前処理/後処理
### 前処理
前処理/後処理はいずれも「LB 35.9 Ensembling & Post Processing Baseline」との差分です（ここでは差分のみを記載）。

- 分数/小数の正規化を拡張（baseline の `_EXACT_FRAC_RE` + `_canon_decimal(float)` を置き換え）:
  - `1/2` や `2 1/2` のような分数表現を Unicode 分数（`½` など）へ変換
  - 小数は「4桁で切り捨てた表現」をキーに Unicode 分数へ寄せる（例: `0.16666...` → `⅙`）
  - `2.5` のような `.5` は要求によりそのまま維持
  - 長い float は小数点以下 4 桁へ短縮（丸めではなく切り捨て）
- 対応する分数の範囲を拡大（baseline より多くの vulgar fraction を扱う）

### 後処理
後処理/後段の提出整形も 「LB 35.9 Ensembling & Post Processing Baseline」との差分です（ここでは差分のみを記載）。

- `5/12 shekel(s)` の置換を修正: baseline の `⅔ shekel 15 grains` → `⅓ shekel 15 grains`
- 分数/小数の正規化を前処理と同じロジックへ統一（`1/2` / `2 1/2` / 小数 → Unicode 分数）
- スラッシュによる選択肢表現の除去を改善:
  - `you / she` のような場合は左側を残す（`you` を残して `she` を落とす）
  - `1/2` のような分数表現を壊さない正規表現に変更
- curly quotes（“ ” ‘ ’）は削除せず、straight quote（`"` / `'`）へ正規化（baseline は削除）
- 禁止文字削除の対象を調整（baseline と違い `()` を一括削除しない）
- stray marks を追加で除去（`..` / `xx...` 等）し、念のため `ḫ/Ḫ` を `h/H` に正規化
- 安全策（提出直前）:
  - 生成が空/None の場合は `<gap>` に強制
  - `validate_submission` で列/行数/空翻訳/重複 id をチェック（baseline は empty 件数のログ出力が中心）
