# train.csv 品質改善プレイブック（半自動・Human-in-the-loop）

## 背景

`train.csv`（1561ペア）は、OCR/転記由来の誤りが混入している可能性がある。具体例:

- 英訳の**途中欠落/途中で切れている**（truncation）
- 原典と**別文が紐づいている**（取り違え）
- 単語レベルの**転記ミス/脱落**
- **数値表記**の不整合（小数の丸め、分数→小数化など）
- 欠損表現の不整合（`<gap>` に寄っていない、表記揺れ）
- PDF上の **italic** 等のスタイル情報が平文化で落ちる

これらは学習のラベルノイズとしてスコアに直撃しやすい一方、1561件すべてを手作業で直すのは重い。ここでは「AI + システム化」で **“怪しいところから優先的に”** 直すフローを定義する。

## 大枠の考え方（やる順番）

1. **自動スキャンで疑わしい行を順位付け**（まずはレビューキューを作る）
2. **原典テキストの取得**（可能ならOCRではなくPDFテキスト抽出）
3. **行↔原典箇所の自動対応付け**（候補を機械で出す）
4. **差分を見せて人が最終確定**（Human-in-the-loopでコストを下げる）
5. 直したデータは **「そのまま学習」だけでなく**、まずは「除外/重み下げ」も選択肢にする

## 1) まずは “怪しい行” を自動で見つける（PDF不要）

目的は「全部を直す」ではなく **“直す価値が高い順”** を作ること。

- 実装: `scripts/scan_train_quality.py`
- 出力: `reports/train_quality_scan.csv`（想定）

検出の例（ヒューリスティック）:

- 英訳の末尾が句読点で終わらない、括弧が閉じていない、極端な長さ比（入力/出力の不自然）
- 分数表記（例: `13 1/3`, `1/2`）や小数の桁数過多（例: `0.33333`）の検出
- `[...]`, `…`, `[x]` など **ギャップっぽい表現**があるのに `<gap>` がない（または `<gap>` の表記揺れ）

この段階で「上位N件だけ」目視して、現場感（どのルールが効く/誤検知が多い）を掴む。

## 2) OCRの前に “PDFテキスト抽出” を試す（切り抜き地獄を避ける）

PDFが **テキストレイヤを持つ** 場合は、切り抜き・OCRよりはるかに楽。

- 代表的手段
  - `pdftotext`（poppler）
  - PyMuPDF（`fitz`）でページ単位抽出

スキャン画像PDFの場合は、先に `ocrmypdf` で **検索可能PDF** を作ってから同じ流れに乗せる（=「毎レコードの切り抜き」を避ける）。

## 3) 行↔原典箇所の自動対応付け（候補を機械で出す）

完全自動は難しいが、「候補ページを出す」だけでも工数が激減する。

使える手がかり:

- `published_texts.csv` の `label`（例: `AKT 6c 705` のような出版物参照）
- `published_texts.csv` の `online transcript`（OAREの当該ページURL）
- `publications.csv`（PDF名+ページ単位のOCRテキスト）を **全文検索インデックス**として使う

実務の型:

1. `label` から「どの出版物（pdf）か」を推定（ファイル名/書誌の正規表現マッチ）
2. そのPDFの `page_text` を対象に、
   - `transliteration` の特徴的なサブストリング
   - `translation` の固有名詞・数値・決まり文句
   をキーにして fuzzy 検索し、上位数ページを候補に出す
3. 候補ページが出たら、当該ブロック（行）を抽出して `train.csv` と diff

## 4) “修正” は AI に提案させ、人間が確定する（UIが効く）

手作業の辛さは「探す」「比較する」「手で打つ」なので、ここを機械に寄せる。

- 構成例（最小）
  - 画面に `transliteration` / `translation` / 原典抽出テキスト（候補1〜3）を並べる
  - 文字差分（diff）をハイライト
  - ワンクリックで「候補を採用」「一部だけ採用」「手編集」

LLMの使いどころ（ローカル実行/オフライン制約に注意）:

- 原典テキスト（PDF抽出/OCR）と `train.csv` を与えて、**差分ベースで**「どこが欠けているか」「数値をどう正規化すべきか」を提案させる
- ただし **最終確定は人間**（取り違えの修正は、間違うと致命傷になりやすい）

## 5) 直す以外の戦略（まずは “除外/重み下げ” で勝てることが多い）

完全修正は重いので、次の段階的戦略が現実的。

- スキャンで高スコア（怪しい）な行を **学習から除外**してCVが上がるかを確認
- さらに余力があれば、上位から順に修正して再学習
- 取り違え疑い（別文ペア）は、直せないなら **除外**が安全（誤ラベルよりマシ）

## ルール例（数値/ギャップの正規化方針のたたき台）

実データに合わせて要調整（まずは “検出だけ” でも価値がある）。

- 分数 `a/b` または `n a/b` の検出 → 変換候補を提示（例: `13 1/3`→`13.3333` など）
- 小数の桁数が多すぎる場合（例: 小数点以下 >4桁）→丸め候補を提示
- ギャップ表現（`[...]`, `…`, `[x]` 等）→ `<gap>` への置換候補を提示
